BITSAI-CR: Automated Code Review via LLM in Practice

BITSAI-CR框架

核心是两大组件：一个是审查评论生成流水线和一个用于持续优化的数据飞轮 (Data Flywheel) 机制

  • comment Generation

    • RuleChecker
      使用经过微调的大型语言模型（LLM），结合219条审查规则的详细分类，首先识别代码中的潜在问题。这些规则包括代码缺陷、性能问题、安全漏洞等方面。
    • ReviewFilter
      在RuleChecker阶段后，ReviewFilter进一步验证并筛选这些发现的潜在问题，确保生成的评论具备较高的精度，避免错误信息或无关评论的生成。
    • Context Preparation：该模块负责准备代码的上下文信息，包括对代码的分块、注释和语法处理，确保后续阶段能够得到清晰的输入。
    • Comment Aggregation：多个类似的评论会被合并，通过余弦相似度算法去重，防止开发者收到重复的反馈。

  • Data Flywheel

    • 实时反馈循环：开发者的反馈会被收集并反馈到系统中，用于更新训练数据集，提升后续审查的准确性和实用性。
    • Outdated Rate（过时率）指标：为了衡量审查评论的实际接受度，论文提出了Outdated Rate指标，即追踪哪些评论未被开发者采纳，如何影响未来的审查精度。这一指标可以有效帮助判断评论是否有实际意义，进而推动模型和规则的优化。

CodeAgent: Autonomous Communicative Agents for Code Review

  • CodeAgent 系统： 集成了多种角色，包括代码提交者、审查员、决策者等，通过角色之间的交互实现全面的代码审查。
  • QA-Checker 代理： 确保审查过程中各代理之间的对话保持一致，并防止主题漂移（即多代理系统中常见的问题），使每个对话都紧扣原始任务目标。
  • 多阶段流水线： CodeAgent 的工作流被分为多个阶段：基础信息同步、代码审查、代码对齐和文档生成，每个阶段都有不同的角色和任务。这些阶段通过精心设计的角色卡（Role Cards）和对话流程来实现。

QLPro: Automated Code Vulnerability Discovery via LLM and Static Code Analysis Integration

该篇论文主要探讨了如何讲llm与传统的静态分析工具（codeql）进行结合

Extraction：负责从海量的项目代码中，找出所有可能与漏洞相关的函数或API，主要是利用了codeql工具来完成这个模块。
Classification and Pairing：此模块的核心任务就是从上一步提取出的海量API中，精准地识别出哪些是Source，哪些是Sink，然后将它们正确地配对起来，形成一个个（Source, Sink）元组。，除此之外还利用了三重投票机制，直接把一大堆API信息丢给LLM，它可能会受到“上下文干扰”，所以此篇文站设计了“三重投票机制”。他们将每个API放在三个完全不同的上下文中，让LLM独立判断三次。最后，根据“少数服从多数”的原则，来决定这个API的最终分类。
Rule Generation：QLPro需要根据上一步生成的（Source, Sink）配对，自动编写出CodeQL可以执行的查询规则，由于codeql的语法比较复杂，没有经过专门的训练的大模型，直接生成语法完全正确的QL代码也是非常困难的，所以此篇文章主要设计了三个角色 作者（Writer, R_w）：这个角色负责根据（Source, Sink）元组，生成初版的QL规则文件 。执行者（Execute, R_e）：它会尝试编译并执行“作者”生成的QL文件 。修复者（Repairer, R_r）：如果编译失败，“执行者”会把失败的QL文件和错误信息一起交给“修复者” 。修复者会分析错误原因，并提出修改建议和解决方案，然后反馈给“作者” 。

相关的工作：Li, Z., Dutta, S., & Naik, M. (2025). IRIS: LLM-assisted static analysis for detecting security vulnerabilities. In The Thirteenth International Conference on Learning Representations
（用预定义的模板来辅助LLM生成CodeQL代码，这种方法的优点是简单、生成代码的格式可控，但缺点是灵活性差，无法应对复杂的代码逻辑和项目特有的API）

Future：
融合动态分析技术：QLPro本质上仍属于静态分析的范畴。静态分析的固有缺陷是可能产生误报，因为它无法获知代码在运行时的真实状态。未来的研究可以探索将QLPro与动态分析（相结合。例如，用QLPro生成的规则先静态地发现潜在漏洞点，然后自动生成Fuzzing的测试用例，对这些可疑点进行动态验证，从而大大提高发现漏洞的准确性。

漏洞修复的自动化：在发现漏洞之后，利用LLM自动生成修复代码意见。

REPOAUDIT: An Autonomous LLM-Agent for Repository-Level Code Auditing

  • Initiator：它根据要查找的漏洞类型，使用轻量级的tree-sitter解析库在整个代码库中快速定位“源头（sources）” 。例如，在检测空指针解引用（NPD）时，它会找出所有赋值为NULL的地方 。

  • Explorer：它不像传统工具那样穷举，而是进行迭代式、按需的探索 ，每次只分析一个函数

    • **程序抽象 (Program Abstraction)**：忽略无关代码，只保留与追踪目标（如NULL值传播）相关的语句 。
    • **指针处理 (Pointer Handling)**：准确判断指针在不同路径下可能指向的对象。
    • **可行路径探索 (Feasible Program Path Exploration)**：识别并只分析逻辑上可能被执行到的路径
      • **智能体记忆 (Agent Memory)**：LLM分析完一个函数后，会将得到的“数据流事实”（data-flow facts）存储在记忆中 。
    • 跨函数跳转：如果在当前函数分析中，发现目标值（如NULL）通过参数传递给了其他函数，或作为返回值返回给了调用者，探索者就会查询预先构建的调用图，然后将分析的焦点跳转到相应的被调用函数或调用者函数，开始新一轮的分析
    • 缓存机制：为了避免重复分析，探索者会利用记忆作为缓存。如果一个函数和特定值的组合已经被分析过，就直接使用结果。

  • Validato：检查LLM生成的数据流事实是否符合代码的控制流顺序，在生成跨函数的漏洞报告时，验证者会再次调用LLM，检查构成整个漏洞链的各个路径片段的执行条件之间是否存在矛盾。
